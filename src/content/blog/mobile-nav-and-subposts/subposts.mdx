子文章这件事，最让我满意的地方是它没有额外引入一套复杂配置。主文章和子文章之间的关系，直接从文件路径里推导出来就够了。

## 关系判断其实很简单

只要一篇文章的 `id` 里包含 `/`，它就不是顶层文章，而是某篇主文章下面的子文章。

```typescript title="核心判断"
export function isSubpost(postId: string): boolean {
  return postId.includes('/')
}

export function getParentId(subpostId: string): string {
  return subpostId.split('/')[0]
}
```

这个规则的好处是足够稳定，也足够直观。你不需要额外写 `parentId`、`series` 或其他 frontmatter 字段，只要文件结构对了，层级关系就自动成立。

## 导航逻辑需要跟着变化

一旦有了子文章，上一篇/下一篇就不再只是线性关系。主文章和子文章应该有不同的导航语义。

<Callout variant="example" title="三种导航上下文">

  2. 读子文章时，在同级子文章之间切换。

</Callout>

## 列表页为什么不展示子文章

我希望博客列表保持清爽，所以列表页只展示主文章。子文章应该是“进入文章之后的结构化延展”，而不是把首页和归档页塞满一串细碎条目。

这也是子文章和“系列文章”最大的区别之一。系列通常在列表页就是多篇平级文章；子文章则更像一本书里的章节，它们服务的是主文章，而不是与主文章争夺入口。

## 这种结构适合哪些内容

它特别适合下面这些类型：

- 一个主题下分多个步骤的教程。
- 需要总览页和细分章节的项目复盘。
- 旅行、读书或研究日志这类天然分章节的长内容。

当你的文章已经出现“必须分段，但拆成独立列表又太碎”的信号时，子文章通常就是更舒服的组织方式。
