移动端目录真正难的地方，不是把一个 `<details>` 放到顶部，而是让它在滚动过程中“有判断力”。用户需要知道自己读到哪一节、还剩多少内容，以及展开目录后应该优先看到哪些条目。

## 置顶区域需要显示什么

我希望这个组件在收起状态下也有足够信息量，所以最终保留了三样东西：章节名称、展开箭头和圆形阅读进度。这样即使用户不点开目录，也能大概知道自己正处在哪一段。

## 当前章节如何判断

最直接的想法通常是监听标题元素是否进入视口，但这会遇到一个问题：当标题已经滚出屏幕、正文内容还在继续时，目录会失去“当前位置”的判断依据。

<Callout variant="example" title="一个常见场景">
  假设一节正文有几百行内容，标题早就滚上去了，但读者显然仍然停留在这一节里。只看标题是否可见，会让目录判断失真。
</Callout>

所以更稳妥的做法，是把每个标题对应到一段“负责范围”：从它自己的位置开始，到下一个标题出现为止。只要当前视口落在这个范围内，就认为这节内容仍然是活跃状态。

```javascript title="思路示例"
function buildHeadingRegions(headings) {
  return headings.map((heading, index) => {
    const nextHeading = headings[index + 1]
    return {
      id: heading.id,
      start: heading.offsetTop,
      end: nextHeading ? nextHeading.offsetTop : document.body.scrollHeight,
    }
  })
}
```

## 为什么要展示多个章节

在一些边界位置，屏幕里可能同时出现上一节的尾巴和下一节的开头。与其强行只高亮一个章节，不如允许目录同时显示多个可见章节，这样读者得到的信息会更真实，也更符合实际阅读状态。

## 体验细节比功能名更重要

很多时候，真正让组件变得“好用”的不是大功能，而是几个小细节：

- 展开后自动滚动到当前活跃条目附近。
- 滚到顶部和底部时，目录区域的遮罩渐变能自然消失。
- 页面切换后重新初始化状态，而不是沿用上一页残留的滚动位置。

这些细节单看都不复杂，但累加起来，才会让移动端目录从“能用”变成“顺手”。
