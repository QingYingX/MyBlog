## 为什么会走到 patch 这一步

做博客模板时，代码高亮通常不是“能不能用”的问题，而是“够不够顺手”。块级代码高亮、内联代码高亮、折叠、标题栏、行号，这些需求单拆出来都不复杂，但组合在一起时，往往会遇到生态层面的缝隙。

这次更新里，问题就出在两个插件的职责边界不够理想。一个更适合代码块，一个更适合内联代码，可一旦同时启用，体验就会变得别扭。与其继续等上游，不如先做一个清晰可控的本地补丁。

## 补丁的思路

核心想法很简单：让某个插件只处理它真正擅长的那部分内容，把不该碰的代码块直接绕开。

```diff title="示例补丁"
 function isBlockCode(element) {
-  return element.tagName === 'pre'
+  return false
 }
```

当然，真实项目里的补丁不会只改这一行，但逻辑就是这么直接。先明确问题边界，再尽量用最小改动把行为修正到自己需要的样子。

## 什么时候值得这样做

我一般只在三种情况下愿意给依赖打补丁：

1. 问题已经定位清楚，而且短期内确实阻塞当前工作。
2. 改动足够小，未来升级时很容易重新验证。
3. 团队知道这里有 patch，不会把它误当成“官方行为”。

如果只是为了偷懒，patch 往往会变成新债；但如果它是一个明确、可追踪、可替换的临时方案，那它反而能帮你把节奏稳住。

## 最后还是要回到长期方案

打补丁的目的不是永远维护一份分叉，而是在“今天必须把事情做完”的前提下，给自己争取一个可用版本。等上游修复、方案更成熟，或者我们自己找到更好的替代品时，再把这段补丁移除，才是更健康的路径。
